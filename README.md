# تحويل البرنامج إلى نظام مركزي (خادم + قاعدة بيانات + تسجيل دخول)

## الفكرة باختصار
البرنامج كان يخزّن كل شيء في `localStorage` (كل جهاز يحتفظ ببياناته لحاله).
الآن أضفنا خادماً (Backend) صغيراً بلغة Node.js وقاعدة بيانات PostgreSQL،
وعدّلنا مكان واحد فقط في الواجهة (`window.storage`) ليتصل بهذا الخادم
بدل `localStorage`. **باقي الـ 15,000 سطر من منطق البرنامج لم يتغيّر إطلاقاً.**

كل عملية حفظ (عميل جديد، دفعة، حركة خزنة...) الآن تُرسَل للخادم، فيخزّنها
في قاعدة بيانات واحدة يشوفها كل أعضاء الفريق من أي جهاز، بعد تسجيل دخول
بحساب خاص بكل واحد منهم.

---

## ⚠️ نقطة مهمة يجب معرفتها (وليست خطأً في التنفيذ)
البيانات ما زالت تُخزَّن على شكل "حزمة واحدة كاملة" لكل جدول (كل بيانات
العملاء مثلاً = قيمة واحدة كبيرة)، تماماً كما كانت في localStorage —
لم نُعِد بناء قاعدة البيانات لتكون صفاً منفصلاً لكل عميل/حركة. لذلك:

- **الاستخدام العادي لفريق من 2-5 أشخاص آمن تماماً.**
- لكن لو **شخصان عدّلا شيت العملاء في نفس اللحظة تقريباً** (خلال نفس
  الثواني)، النظام يكتشف هذا التعارض ويرفض حفظ التعديل الثاني بدل الكتابة
  فوق الأول بصمت (رسالة "⚠️ تعارض... يرجى تحديث الصفحة")، بدل ضياع بيانات
  بصمت. الحل وقتها: يحدّث الشخص الثاني الصفحة (F5) ويعيد تنفيذ عمليته.
- إذا لاحقاً احتجتم أداءً أعلى مع تعديلات متزامنة كثيفة جداً (فريق أكبر
  من 5-10 يعدّلون نفس الشيت باستمرار)، الخطوة التالية المنطقية هي تحويل
  كل جدول (عملاء، حركات، حقائب...) إلى صفوف منفصلة في قاعدة البيانات
  بدل حزمة واحدة — وهذا مشروع إضافي منفصل يمكن تنفيذه تدريجياً لاحقاً.

## الملفات
```
server/     ← الخادم (Node.js + Express + PostgreSQL)
  server.js                     نقطة تشغيل الخادم + كل الـ API
  db.js                         الاتصال بقاعدة البيانات
  auth.js                       تسجيل الدخول والتشفير
  schema.sql                    تعريف الجداول
  seed-user.js                  أداة لإضافة مستخدم جديد للفريق
  migrate-from-localstorage.js  أداة نقل بياناتكم الحالية للخادم
  package.json / .env.example

frontend/
  app.html    ← نفس برنامجكم، بعد تعديل نقطة التخزين + شاشة دخول جديدة
```

## خطوات التشغيل

### 1) إنشاء قاعدة بيانات واستضافة (Render — الأسهل لفريق صغير)
1. أنشئ حساباً مجانياً على **render.com**.
2. من لوحة التحكم: **New → PostgreSQL** → أعطها اسماً → أنشئها، ثم انسخ
   قيمة **Internal Database URL** (تبدأ بـ `postgres://...`).
3. من لوحة التحكم: **New → Web Service** → اربطه بمستودع الكود (ارفعوا
   مجلد `server/` و`frontend/` إلى GitHub أولاً)، أو استخدموا خيار الرفع
   المباشر للملفات إن توفر.
4. في إعدادات الـ Web Service، أضف متغيرات البيئة (Environment):
   - `DATABASE_URL` = الرابط اللي نسخته في الخطوة 2
   - `JWT_SECRET` = نص عشوائي طويل (ولّده مرة واحدة ولا تغيّره بعدها)
5. Build Command: `cd server && npm install`
   Start Command: `cd server && node server.js`
6. بعد نجاح النشر، ستحصلون على رابط عام مثل:
   `https://your-app-name.onrender.com` — هذا هو "الويب سايت" الجديد.

> **بديل:** Railway.app بنفس الخطوات تقريباً (قاعدة بيانات Postgres +
> خدمة Node)، وهو خيار جيد آخر لو Render ما ناسبكم.

### 2) إضافة حسابات الفريق (2-5 أشخاص)
من جهازك، بعد رفع الخادم وتشغيله، شغّل محلياً (يحتاج نفس `DATABASE_URL`
في ملف `.env` عندك):
```
cd server
npm install
node seed-user.js ahmed "كلمة_مرور_قوية" "أحمد"
node seed-user.js sara  "كلمة_مرور_قوية2" "سارة"
```
كرّر السطر لكل عضو بالفريق. كل واحد يسجّل دخول لاحقاً باسم المستخدم
وكلمة المرور اللي حددتها له.

### 3) ترحيل بياناتكم الحالية (آلاف العملاء والحركات)
1. افتح البرنامج **القديم** (النسخة اللي تعمل بـ localStorage) في المتصفح.
2. اضغط F12 (أدوات المطوّر) → تبويب Console → الصق الكود الموجود في
   أعلى ملف `migrate-from-localstorage.js` واضغط Enter. سيُنزَّل ملف
   `exported-data.json`.
3. انسخ الملف بجانب `server/` وشغّل:
   ```
   node migrate-from-localstorage.js ./exported-data.json
   ```
4. تأكد إن الرسالة الأخيرة "✅ اكتمل الترحيل" ظهرت لكل المفاتيح.

### 4) الاستخدام اليومي
كل عضو فريق يفتح رابط الموقع (`https://your-app-name.onrender.com`) من
أي جهاز/متصفح → يدخل كود الترخيص (أول مرة فقط) → يسجّل دخول بحسابه على
الخادم (من الخطوة 2) → ثم يسجّل دخول بحسابه الداخلي المعتاد في البرنامج
(شيت المستخدمين الموجود مسبقاً — بدون تغيير).

## ملاحظات أمان
- لا تشارك ملف `.env` أو قيمة `JWT_SECRET` مع أي أحد خارج الفريق.
- الخادم لا يفكّ تشفير أي بيانات إطلاقاً — البيانات مشفّرة من المتصفح
  قبل إرسالها، وتبقى مشفّرة داخل قاعدة البيانات، تماماً كما كانت راحة
  التشفير الحالية في البرنامج.
- يُفضّل تفعيل النسخ الاحتياطي التلقائي لقاعدة البيانات من لوحة تحكم
  Render/Railway (متوفر غالباً حتى في الخطط المجانية/الرخيصة).
